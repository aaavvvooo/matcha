<overview>

## Problem Statement
Young adults seeking romantic connections need a reliable, safe platform to discover and communicate with compatible partners nearby. Existing solutions are either too casual (swipe-only, no depth) or too complex (subscription walls, cluttered UX). Matcha is a full-featured dating web application where users discover profiles based on geographic proximity, shared interests, and compatibility — and can chat in real time once they mutually connect.

The platform is currently functional only at the authentication layer. All core dating features (profile management, discovery, social mechanics, real-time communication) remain unimplemented, blocking any meaningful use.

## Target Users
- **Primary**: Adults aged 18–65 seeking romantic or social connections, comfortable using browser-based apps.
- **Secondary**: 42 school evaluators verifying subject compliance (minimum 500 profiles, all mandatory features demonstrable).

## Success Metrics
- User can complete full registration → profile setup → browse → like → mutual match → chat flow end-to-end with zero server errors.
- Browse endpoint returns orientation-filtered, distance-sorted results for any authenticated user with a complete profile.
- Real-time chat message delivered within 10 seconds of sending.
- Minimum 500 seed profiles present in the database for evaluation.
- All mandatory security requirements pass (no SQL injection, no XSS, MIME-validated uploads, rate-limited sensitive endpoints).

</overview>

<functional-decomposition>

## Capability Tree

### Capability: Profile Management
Users can create and maintain a rich public profile including personal info, photos, location, and interest tags.

#### Feature: Profile Setup (First-Time)
- **Description**: New users complete their profile before they can browse or be discovered.
- **Inputs**: first_name, last_name, gender, sexual_preference, biography, at least 1 photo, at least 1 tag, location.
- **Outputs**: Populated user_profiles row; user redirected to /browse.
- **Behavior**: All fields validated server-side; profile marked complete only when photo + tag + location are set; incomplete profiles are redirected to /profile/setup by middleware.

#### Feature: Profile Edit
- **Description**: Authenticated users update any profile field at any time.
- **Inputs**: Partial update payload (any subset of profile fields).
- **Outputs**: Updated user_profiles row; updated response payload.
- **Behavior**: PATCH semantics; only provided fields are changed; email uniqueness re-validated if changed.

#### Feature: Photo Upload and Management
- **Description**: Users upload up to 5 photos; one is designated as the profile picture.
- **Inputs**: Multipart image file (MIME validated), photo ID for delete/set-main.
- **Outputs**: File stored in /uploads volume; photos row created with URL and order.
- **Behavior**: MIME type checked with python-magic (not extension); max 5MB; max 5 photos; filename replaced with UUID; first upload auto-sets is_main=true; cannot delete last photo if it is the profile picture.

#### Feature: Tag Management
- **Description**: Users associate interest tags (reusable pool) with their profile.
- **Inputs**: List of tag names (new or existing).
- **Outputs**: Tags created if new; user_tags rows updated (replace full list).
- **Behavior**: Tag names stripped of HTML; GET /tags returns full list for autocomplete; POST /profile/tags replaces existing associations atomically.

---

### Capability: Geolocation
The system captures and maintains each user's geographic position to power distance-based discovery.

#### Feature: Browser GPS Consent
- **Description**: On profile setup, request GPS coordinates from the browser with explicit user consent.
- **Inputs**: Browser geolocation API response (latitude, longitude).
- **Outputs**: lat/lon stored in user_profiles; location_label resolved and stored.
- **Behavior**: If consent granted → send coords to backend directly. If denied → show city text input → geocode via Nominatim API → send resolved coords.

#### Feature: Manual Location Update
- **Description**: Users update their location at any time from profile settings.
- **Inputs**: Text city input or new GPS coordinates.
- **Outputs**: Updated lat/lon/location_label in user_profiles.
- **Behavior**: Same geocoding flow as initial setup; triggers fame recalculation if relevant.

#### Feature: Distance Calculation
- **Description**: Compute geographic distance between two users for sorting and filtering.
- **Inputs**: Two (lat, lon) pairs.
- **Outputs**: Distance in kilometers (float).
- **Behavior**: Haversine formula implemented as a SQL expression or Python utility; whitelisted sort column name injection prevention.

---

### Capability: Discovery and Search
Authenticated users with complete profiles can browse and search for compatible matches.

#### Feature: Browse Feed
- **Description**: Returns a paginated list of compatible profiles sorted by relevance.
- **Inputs**: Authenticated user context; optional query params: sort, age_min, age_max, fame_min, fame_max, tags, page, limit.
- **Outputs**: Array of profile cards (id, first_name, age, profile_picture_url, fame_rating, distance_km, shared_tags, is_online, last_seen).
- **Behavior**: Filter by sexual orientation compatibility; exclude blocked users and own profile; sort by distance → shared tags → fame rating; apply optional user filters; paginate.

#### Feature: Advanced Search
- **Description**: User-driven search with explicit multi-criteria filtering.
- **Inputs**: age_min, age_max, fame_min, fame_max, lat, lon, radius_km, tags (AND logic), sort, page.
- **Outputs**: Same profile card array as browse.
- **Behavior**: Reuses browse query with different defaults; location search applies Haversine ≤ radius filter; orientation and block exclusion still applied.

---

### Capability: Social Mechanics
Users interact with profiles via likes, views, blocks, and reports, forming the connection layer that enables chat.

#### Feature: Like / Unlike
- **Description**: Users like or unlike a profile; mutual likes create a connection.
- **Inputs**: Authenticated user, target profile ID.
- **Outputs**: Like record created/removed; connection event if mutual; notifications dispatched; fame recalculated for target.
- **Behavior**: Liker must have a profile picture (403 otherwise); unique constraint on (liker_id, liked_id); mutual like detection triggers both users notified; unlike removes connection and disables chat.

#### Feature: Profile View Recording
- **Description**: Visiting a profile is logged and shown to the viewed user.
- **Inputs**: Authenticated viewer, target profile ID.
- **Outputs**: profile_views row inserted; fame recalculated for viewed user; notification dispatched.
- **Behavior**: Do not record a view when user visits their own profile; all views kept (not deduplicated).

#### Feature: Block and Report
- **Description**: Users can block or report other users for safety.
- **Inputs**: Authenticated user, target profile ID, optional report reason.
- **Outputs**: blocks/reports row created; blocked user removed from all future browse/search results for both parties; any existing like removed on block.
- **Behavior**: Unique constraint on (blocker_id, blocked_id); reports stored for admin review; report decrements fame of reported user.

#### Feature: Profile View History and Liked-By Lists
- **Description**: Users see who has viewed or liked their profile.
- **Inputs**: Authenticated user (own profile context).
- **Outputs**: Paginated lists with viewer/liker info and timestamps.
- **Behavior**: Only own data accessible; sorted by most recent.

---

### Capability: Fame Rating
A dynamic score (0–100) representing each user's popularity and profile quality, used as a browse/search sort criterion.

#### Feature: Fame Calculation
- **Description**: Recalculates a user's fame rating based on social interactions.
- **Inputs**: user_id; counts of likes received, views received, connections, blocks received, reports, profile completeness.
- **Outputs**: Updated fame_rating in user_profiles.
- **Behavior**: Formula: +10/like received (max 50), +5/view (max 30), +15 for complete profile, +5/connection, -15/block received, -20/report (max -40). Capped 0–100. Called on every relevant event; also run as hourly Celery periodic task as safety net.

---

### Capability: Online Status
The system tracks and exposes whether each user is currently active.

#### Feature: Online Presence Tracking
- **Description**: Mark a user online when WebSocket connected; update last_seen on activity.
- **Inputs**: WebSocket connect/disconnect events; heartbeat pings every 60s.
- **Outputs**: Redis key `user:{id}:online` with 300s TTL; `users.last_seen` timestamp updated.
- **Behavior**: On connect → SET Redis key. On disconnect → DELETE key. Heartbeat refreshes TTL. last_seen updated on every auth action and heartbeat.

#### Feature: Online Status on Profile
- **Description**: Any profile card or view page displays whether the user is online or shows last seen time.
- **Inputs**: target user_id.
- **Outputs**: `is_online` (bool), `last_seen` (datetime) included in profile response.
- **Behavior**: Check Redis key for is_online; fall back to users.last_seen for display.

---

### Capability: Real-Time Chat
Mutually connected users exchange text messages with sub-10-second delivery.

#### Feature: WebSocket Connection Management
- **Description**: Maintain authenticated WebSocket connections per user for real-time communication.
- **Inputs**: Access token (query param on WS URL).
- **Outputs**: Persistent bidirectional channel; connection registered in ConnectionManager.
- **Behavior**: Validate token on upgrade; store {user_id: WebSocket} in ConnectionManager; handle disconnect gracefully; client reconnects automatically.

#### Feature: Message Sending and Delivery
- **Description**: Send a chat message from one connected user to another; persist and deliver in real time.
- **Inputs**: Sender WebSocket, JSON payload {type: "message", to_user_id, content}.
- **Outputs**: Message persisted to messages table; delivered to recipient WebSocket if online; stored for retrieval if offline.
- **Behavior**: Validate mutual connection (both liked, no block); reject if connection broken; index on (sender_id, receiver_id) for history queries.

#### Feature: Chat History
- **Description**: Fetch paginated message history between two users.
- **Inputs**: Authenticated user, other user_id, page, limit.
- **Outputs**: Paginated list of messages with sender, content, sent_at, read_at.
- **Behavior**: Only accessible to participants of the conversation; auto-scroll to latest on page load.

---

### Capability: Real-Time Notifications
Users receive sub-10-second notifications for key social events from any page on the site.

#### Feature: Notification Dispatch
- **Description**: Server-side notification emission when social events occur.
- **Inputs**: Event type (like, view, message, match, unlike), actor user, recipient user_id.
- **Outputs**: Notification persisted to notifications table; delivered over WebSocket if recipient is online.
- **Behavior**: Multiplexed over same WebSocket as chat using `type` field; events: like created, view recorded, message received, mutual like, like deleted.

#### Feature: Notification REST API
- **Description**: Fetch, display, and mark notifications as read.
- **Inputs**: Authenticated user; optional unread_only filter; notification ID for mark-read.
- **Outputs**: List of notifications; updated is_read status.
- **Behavior**: GET /notifications hydrates badge on page load before WS connects; PATCH /notifications/{id}/read and /notifications/read-all.

---

### Capability: Database Seed
500 realistic profiles prepopulate the database for evaluation and testing.

#### Feature: Seed Script
- **Description**: Generate and insert 500 distinct realistic user profiles with all required data.
- **Inputs**: None (run once via Docker Compose or Makefile).
- **Outputs**: 500 users with profiles, photos, tags, locations distributed across 5–10 cities.
- **Behavior**: Use Faker for names/bios/emails; randomized gender/preference distributions; real lat/lon for cities; 1–5 tags from predefined list of ~30; photo URLs from picsum.photos; fame_rating 0–100 random; is_validated=True; skip if profiles already exist.

---

### Capability: Security Hardening
Mandatory security requirements from the project subject.

#### Feature: Input Sanitization
- **Description**: Prevent XSS and HTML injection in all user-supplied text fields.
- **Inputs**: Any text field (bio, tags, usernames, etc.).
- **Outputs**: Sanitized string stored in DB.
- **Behavior**: Pydantic validators strip/reject HTML tags on all text inputs; React default escaping; never use dangerouslySetInnerHTML.

#### Feature: SQL Injection Prevention
- **Description**: All database queries use parameterized statements.
- **Inputs**: Any user-supplied filter values (sort fields, search criteria).
- **Outputs**: Safe parameterized query execution.
- **Behavior**: SQLAlchemy parameterized expressions everywhere; sort column names whitelisted server-side before interpolation.

#### Feature: Secure File Upload
- **Description**: Validate uploaded files are genuine images, reject malicious content.
- **Inputs**: Uploaded file bytes.
- **Outputs**: Accepted (stored) or rejected (400 error).
- **Behavior**: python-magic MIME check (not extension); whitelist: image/jpeg, image/png, image/gif, image/webp; max 5MB; UUID-renamed filename; stored outside web root if possible.

---

### Capability: Bonus Features
Optional enhancements evaluated only after all mandatory features are complete.

#### Feature: OmniAuth / Social Login
- **Description**: Register and login via OAuth providers (Google, GitHub, 42 intra).
- **Inputs**: OAuth provider callback with code and state.
- **Outputs**: User created/linked; session established.
- **Behavior**: Authlib for OAuth flow; oauth_accounts table; if email matches existing user → link; else create new user with is_validated=True.

#### Feature: Drag-and-Drop Photo Gallery with Editing
- **Description**: Reorder photos via drag-and-drop; crop/rotate/filter before upload.
- **Inputs**: Drag-and-drop events; canvas-edited image blob.
- **Outputs**: Updated photo order in DB; edited image file uploaded.
- **Behavior**: react-beautiful-dnd or @dnd-kit for reordering; react-image-crop for crop; canvas for rotation/filters; PATCH /profile/photos/reorder for order update.

#### Feature: Interactive Map View
- **Description**: Map page showing nearby users as pins with profile pictures.
- **Inputs**: User location; browse results with lat/lon.
- **Outputs**: Leaflet map with user markers; popup with name/age/fame linking to profile.
- **Behavior**: React-Leaflet; navigator.geolocation.watchPosition() for real-time tracking; same orientation/block filtering as browse.

#### Feature: Video / Audio Chat
- **Description**: Connected users initiate peer-to-peer video or audio calls.
- **Inputs**: WebRTC offer/answer/ICE signals over existing WebSocket.
- **Outputs**: Live media stream between peers.
- **Behavior**: WebRTC + STUN (Google's stun:stun.l.google.com:19302); simple-peer library; signaling over WebSocket; "Video call" button on ChatPage.

#### Feature: Date / Event Scheduling
- **Description**: Connected users propose, accept, and track real-life dates.
- **Inputs**: Title, datetime, location label, recipient user_id.
- **Outputs**: Event record created; notifications dispatched; events list page.
- **Behavior**: events table with ENUM status (pending/accepted/declined/cancelled); proposer can cancel; notifications on proposal and status change; countdown display.

## Module Definitions

### Module: profile (backend)
- **Maps to capability**: Profile Management
- **Responsibility**: All profile CRUD, photo upload/management, tag association
- **File structure**:
  ```
  api/endpoints/profile.py
  service/profile_service.py
  service/photo_service.py
  repository/profile_repo.py
  models/photos.py (extend)
  models/user_profiles.py (extend)
  ```
- **Exports**:
  - `POST /profile/setup` — first-time profile creation
  - `GET /profile/me` — own profile
  - `PATCH /profile/me` — update profile
  - `POST /profile/photos` — upload photo
  - `DELETE /profile/photos/{id}` — remove photo
  - `PATCH /profile/photos/{id}` — set main photo
  - `GET /tags` — list all tags
  - `POST /tags` — create tag
  - `POST /profile/tags` — associate tags

### Module: browse (backend)
- **Maps to capability**: Discovery and Search
- **Responsibility**: Orientation-filtered, distance-sorted discovery and explicit search
- **File structure**:
  ```
  api/endpoints/browse.py
  service/browse_service.py
  utils/haversine.py
  ```
- **Exports**:
  - `GET /browse` — paginated filtered browse
  - `GET /search` — explicit criteria search

### Module: social (backend)
- **Maps to capability**: Social Mechanics + Fame Rating
- **Responsibility**: Likes, views, blocks, reports, fame recalculation
- **File structure**:
  ```
  api/endpoints/social.py
  service/social_service.py
  service/fame_service.py
  repository/social_repo.py
  models/likes.py
  models/profile_views.py
  models/blocks.py
  models/reports.py
  tasks/fame_tasks.py
  ```
- **Exports**:
  - `GET /profile/{id}` — public profile view (records view)
  - `POST /profile/{id}/like` — like
  - `DELETE /profile/{id}/like` — unlike
  - `POST /profile/{id}/block` — block
  - `DELETE /profile/{id}/block` — unblock
  - `POST /profile/{id}/report` — report
  - `GET /profile/me/views` — who viewed me
  - `GET /profile/me/liked-by` — who liked me

### Module: realtime (backend)
- **Maps to capability**: Real-Time Chat + Notifications + Online Status
- **Responsibility**: WebSocket connection management, message routing, notification dispatch
- **File structure**:
  ```
  api/endpoints/websocket.py
  api/endpoints/chat.py
  api/endpoints/notifications.py
  service/websocket_manager.py
  service/chat_service.py
  service/notification_service.py
  repository/chat_repo.py
  repository/notification_repo.py
  models/messages.py
  models/notifications.py
  ```
- **Exports**:
  - `WS /ws` — authenticated WebSocket endpoint
  - `GET /chat/{user_id}/history` — message history
  - `GET /notifications` — list notifications
  - `PATCH /notifications/{id}/read` — mark read
  - `PATCH /notifications/read-all` — mark all read

### Module: seed (backend)
- **Maps to capability**: Database Seed
- **Responsibility**: Generate 500 realistic profiles for evaluation
- **File structure**:
  ```
  scripts/seed.py
  ```
- **Exports**: CLI script (no HTTP interface)

## Dependency Chain

### Foundation Layer (Phase 0)
No dependencies — build these first.

- **db-migrations**: Adds all missing columns and tables (lat/lon/location_label on user_profiles, last_seen on users, likes, profile_views, blocks, reports, messages, notifications, events). All other modules depend on these tables existing.
- **utils-haversine**: Pure function, no dependencies. Distance calculation used by browse, social.
- **utils-sanitizer**: Pure function, no dependencies. Input sanitization used by profile, social.
- **uploads-volume**: Docker volume + static route for /uploads. Photo service depends on this.

### Profile Layer (Phase 1)
Depends on: db-migrations, uploads-volume, utils-sanitizer

- **profile-module**: Depends on [db-migrations, uploads-volume, utils-sanitizer]. Provides profile data that browse and social modules query.
- **geolocation-client**: Depends on [db-migrations]. Nominatim geocoding; used by profile-module.

### Discovery Layer (Phase 2)
Depends on: profile-module, utils-haversine, db-migrations

- **browse-module**: Depends on [profile-module, utils-haversine, db-migrations]. Requires profiles with locations and tags to return meaningful results.

### Social Layer (Phase 3)
Depends on: profile-module, db-migrations, notification-service (async call)

- **social-module**: Depends on [profile-module, db-migrations]. Like/view/block/report actions. Triggers fame recalculation.
- **fame-service**: Depends on [social-module, profile-module, db-migrations]. Recalculates score; called by social-module and periodic Celery task.

### Real-Time Layer (Phase 4)
Depends on: social-module (for connection validation), db-migrations

- **websocket-manager**: Depends on [db-migrations, social-module]. Connection manager must check mutual-like status before routing messages.
- **chat-service**: Depends on [websocket-manager, db-migrations].
- **notification-service**: Depends on [websocket-manager, social-module, db-migrations]. Dispatched by social-module events.
- **online-status**: Depends on [websocket-manager, redis-client]. Redis already exists.

### Seed Layer (Phase 5)
Depends on: All Phase 0–3 modules (needs all tables and models to exist)

- **seed-script**: Depends on [db-migrations, profile-module, social-module]. Inserts data through models, not HTTP.

### Bonus Layer (Phase 6)
Depends on: All mandatory layers complete

- **omniauth**: Depends on [auth-module (existing), db-migrations].
- **photo-gallery-dnd**: Depends on [profile-module (photo reorder endpoint)].
- **map-view**: Depends on [browse-module, geolocation-client].
- **video-chat**: Depends on [websocket-manager, chat-service].
- **event-scheduling**: Depends on [social-module, notification-service, db-migrations].

</dependency-graph>

  - Test strategy: Manual psql inspection.

- [ ] Create Alembic migration: add `last_seen TIMESTAMP` to users (depends on: none)
  - Acceptance criteria: Column visible after migration.

- [ ] Create Alembic migration: create tables `likes`, `profile_views`, `blocks`, `reports`, `messages`, `notifications` with all FK constraints and unique constraints (depends on: none)
  - Acceptance criteria: All tables created; FK integrity enforced.
  - Test strategy: psql \d table_name checks.

- [ ] Create Alembic migration: create `events` table (bonus; depends on: none)

## Critical Test Scenarios

### Profile Management
**Happy path**:
- Complete profile setup with all required fields → 201; profile_complete=true
- Upload valid JPEG under 5MB → photo row created, UUID filename in response

**Edge cases**:
- Upload 6th photo → 400 "max 5 photos"
- Delete only photo when it is profile picture → 400 "cannot delete"
- PATCH with only bio field → only bio updated, other fields unchanged

**Error cases**:
- Upload .php file disguised as .jpg → 400 MIME rejection
- Bio with `<script>alert(1)</script>` → sanitized on save; not stored as HTML
- POST /profile/setup twice → 400 "already set up"

### Browse and Search
**Happy path**:
- Straight male user browses → only female profiles returned
- Bisexual user browses → all genders returned
- Results sorted by distance ascending by default

**Edge cases**:
- User with no location set → excluded from results or shown at end
- All nearby users are blocked → empty result set (not 404)
- search with radius_km=0 → only exact location matches

**Error cases**:
- sort='; DROP TABLE users; -- → whitelisted sort field rejects input → 400

### Social Mechanics
**Happy path**:
- User A likes User B; User B likes User A → both notified of mutual match
- User A blocks User B → B disappears from A's browse; A disappears from B's browse

**Edge cases**:
- User without profile picture tries to like → 403
- User likes same profile twice → 409 or idempotent 200
- View own profile → no view recorded

**Error cases**:
- Like a blocked user → 403
- Deleted user's profile viewed → 404

### Real-Time Chat
**Happy path**:
- User A sends message to connected User B → B receives within 10 seconds
- Page load → last 20 messages fetched and displayed

**Edge cases**:
- User A sends message to User B after unlike → 403 "not connected"
- User A sends message to blocked User B → 403
- Recipient is offline → message stored; delivered when they reconnect and fetch history

**Error cases**:
- WebSocket connection with invalid token → close 4001
- Message content over max length → 400

### Security
**Happy path**:
- All auth endpoints enforce rate limits → 429 after threshold

**Edge cases**:
- XSS in bio field stored and retrieved → HTML stripped, not rendered as markup
- SQL injection in sort parameter → rejected by whitelist → 400

## Test Generation Guidelines
- Use pytest + httpx AsyncClient for integration tests against the FastAPI app.
- Seed test DB with factory functions (not the 500-profile seed script).
- Mock Brevo and Nominatim external API calls with pytest-mock.
- For WebSocket tests, use FastAPI's TestClient WebSocket context manager.
- Never use `time.sleep()` in tests — use asyncio.wait_for with timeout.
- All tests must be isolated — each test rolls back DB changes via transaction rollback or truncation.

</test-strategy>

<architecture>

## System Components

```
Browser (React SPA)
    ↕ HTTPS REST + WSS WebSocket
FastAPI Backend
    ├── Auth Layer (JWT + refresh cookie)
    ├── Profile / Browse / Social / Chat / Notification Endpoints
    ├── WebSocket ConnectionManager (in-process dict)
    └── Celery Task Dispatcher
         ├── Email Tasks → Brevo SMTP API
         └── Fame Recalculation Tasks
PostgreSQL ← SQLAlchemy async (asyncpg)
Redis ← token blacklist + online presence + Celery broker
/uploads volume ← static file storage for photos
```

## Data Models

**Core Entities**: users, user_profiles, photos, tags, user_tags (all exist)

**Social Layer (new)**:
- `likes(id, liker_id FK, liked_id FK, created_at)` — UNIQUE(liker_id, liked_id)
- `profile_views(id, viewer_id FK, viewed_id FK, viewed_at)` — indexed on viewed_id
- `blocks(id, blocker_id FK, blocked_id FK, created_at)` — UNIQUE(blocker_id, blocked_id)
- `reports(id, reporter_id FK, reported_id FK, reason TEXT, created_at)`

**Real-Time Layer (new)**:
- `messages(id, sender_id FK, receiver_id FK, content TEXT, sent_at, read_at nullable)` — indexed on (sender_id, receiver_id)
- `notifications(id, recipient_id FK, type VARCHAR, from_user_id FK, related_id nullable, is_read BOOL, created_at)`

**Bonus**:
- `events(id, proposer_id FK, recipient_id FK, title, description, proposed_datetime, location_label, status ENUM, created_at, updated_at)`
- `oauth_accounts(id, user_id FK, provider VARCHAR, provider_user_id VARCHAR, access_token TEXT)`

**Column additions**:
- `user_profiles`: lat FLOAT, lon FLOAT, location_label VARCHAR(255)
- `users`: last_seen TIMESTAMP

## Technology Stack

**Backend**: FastAPI (Python 3.13), SQLAlchemy 2.0 async, asyncpg, Alembic, Celery + Redis, Argon2, JWT, SlowAPI, python-magic, Faker (seed), Authlib (bonus)

**Frontend**: React 19, React Router v7, Axios, react-hot-toast, React-Leaflet (bonus), react-beautiful-dnd (bonus), react-image-crop (bonus), simple-peer (bonus)

**Infrastructure**: Docker Compose (PostgreSQL, Redis, backend, frontend, Celery worker, Flower)

**Decision: WebSocket for real-time (not polling)**
- **Rationale**: Subject requires ≤10s delivery; WebSocket allows sub-second delivery and reuse for both chat and notifications.
- **Trade-offs**: More complex connection management; must handle reconnect logic client-side.
- **Alternatives considered**: Long polling (simpler but higher latency and server load).

**Decision: Haversine in Python/SQL instead of PostGIS**
- **Rationale**: PostGIS adds deployment complexity; Haversine formula is sufficient for city-level distance calculation.
- **Trade-offs**: Less accurate than geodesic; no advanced spatial indexing.
- **Alternatives considered**: PostGIS with ST_Distance (overkill for this scale).

**Decision: Local /uploads volume instead of S3**
- **Rationale**: No cloud credentials required; simpler for 42 school evaluation environment; subject does not mandate cloud storage.
- **Trade-offs**: Photos lost if volume not persisted; not horizontally scalable.
- **Alternatives considered**: S3/MinIO (better but adds complexity and cost).

</architecture>

<risks>

## Technical Risks

**Risk**: WebSocket authentication via query param (token in URL) logs token in access logs.
- **Impact**: Medium — token exposure in server logs
- **Likelihood**: Low (evaluation environment, short-lived tokens)
- **Mitigation**: Use short-lived access tokens (15 min already configured); clear tokens from logs if possible; alternatively validate token in first WebSocket message instead of URL.
- **Fallback**: Accept the risk for evaluation; document it.

**Risk**: Haversine distance query becomes slow with 500+ profiles at scale.
- **Impact**: Low (evaluation has 500 profiles, not millions)
- **Likelihood**: Low
- **Mitigation**: Add index on (lat, lon) on user_profiles; apply bounding-box pre-filter before Haversine calculation.
- **Fallback**: Compute distances in Python after a bounding-box SQL filter.

**Risk**: MIME type validation with python-magic requires libmagic system dependency in Docker.
- **Impact**: Medium — photo upload broken if not installed
- **Likelihood**: Medium
- **Mitigation**: Add `RUN apt-get install -y libmagic1` to backend Dockerfile.
- **Fallback**: Use filetype Python library which bundles its own magic bytes database.

## Dependency Risks

**Risk**: Nominatim geocoding API has rate limits (1 req/sec, requires User-Agent header).
- **Impact**: Medium — location setup fails under load
- **Likelihood**: Low (individual users on evaluation)
- **Mitigation**: Add proper User-Agent header; cache geocoding results; add retry with backoff.
- **Fallback**: Allow manual lat/lon entry as direct fallback.

**Risk**: Brevo email API credentials not configured in evaluation environment.
- **Impact**: Low — email verification still works (token visible in DB)
- **Likelihood**: Low (already working)
- **Mitigation**: Document .env setup; provide test credentials.

## Scope Risks

**Risk**: Phase 4 (real-time) is the most complex phase and may consume disproportionate time.
- **Impact**: High — bonus features would be cut
- **Likelihood**: Medium
- **Mitigation**: Complete Phases 0–3 first; treat Phase 4 as a unit with clear entry/exit criteria; start with basic WebSocket before notifications.
- **Fallback**: Long-polling fallback for notifications if WebSocket proves unmanageable.

**Risk**: 500-profile seed script requires all tables to exist; running it before migrations complete will fail.
- **Impact**: Medium — evaluation blocked
- **Likelihood**: Low
- **Mitigation**: Seed script is Phase 5 (last before bonus); document run order.
- **Fallback**: Idempotent script can be re-run after fixing migrations.

</risks>

## Glossary
- **Connection**: Two users who have mutually liked each other. Prerequisite for chat.
- **Fame Rating**: A calculated score (0–100) reflecting a user's popularity and profile quality.
- **Profile Complete**: A user profile that has gender, sexual_preference, biography, at least 1 tag, at least 1 photo, and a location set.
- **Orientation Compatibility**: Browse/search filter ensuring users only see profiles relevant to their sexual preference.
- **Seed Profile**: A generated profile used to populate the database for evaluation.
